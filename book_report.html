Research Methodology

This evaluation uses a standardized benchmark of ICD-10 codes from the Centers for Medicare & Medicaid Services (CMS) 2024 dataset. Each code includes an official medical description, which serves as input to the language models. We measure precision, recall, and F1 scores to evaluate model performance in medical coding tasks.

Chapter 2.1: Constrained Prompting Analysis

Introduction: The Hallucination Problem

In baseline experiments (Chapter 2), we observed that models sometimes generate ICD-10 codes with higher specificity than warranted by the input description. For example, given a general description like "Cholera", a model might predict specific subtypes (e.g., A00.1) rather than the general code (A00). This phenomenon—known as hallucination or over-specification—can lead to reduced precision.

Hypothesis

Anti-Hallucination Instructions

We hypothesize that adding explicit anti-hallucination instructions to the system prompt will improve precision by reducing false positives from over-specification. Specifically, we instruct models to:

  • Provide ONLY codes that can be DEFINITIVELY inferred from the description
  • NOT add specificity beyond what is explicitly stated
  • Prefer general codes over specific subtypes when details are absent
  • Avoid false positives by not hallucinating details

Experimental Setup

We created constrained versions of both Claude and Codex models with modified system prompts that include the anti-hallucination instructions above. All other parameters (temperature, dataset, evaluation metrics) remain identical to baseline experiments.

Chapter 3: Bidirectional Consistency Testing

Introduction: Testing Model Understanding

Beyond one-way prediction accuracy, we can test whether models truly understand medical codes by examining bidirectional consistency: Can a model generate a description from a code, then correctly predict that code from its own description?

Experimental Design

Round-Trip Testing Protocol

For each ICD-10 code, we perform a three-step process:

  1. Forward Generation: Code → Description at varying detail levels (0-10)
  2. Reverse Prediction: Description → Predicted Codes
  3. Consistency Check: Does predicted code match original?

We test 11 different description detail levels to identify the minimum specificity needed for accurate round-trip prediction.

Detail Level Variations

Each ICD-10 code is expanded into descriptions at 11 detail levels (0=minimal, 10=maximal). This allows us to analyze how description specificity affects prediction accuracy.

Chapter 3.1: RAG-Enhanced Prediction

The Problem from Chapter 3

Chapter 3 revealed two critical issues when asking models to predict codes from their own generated descriptions:

  • Low Correctness: Only ~30% of predictions matched the actual medical code
  • Poor Consistency: The same code's 11 variants often produced wildly different predictions

The Hypothesis: Can Examples Help?

Chapter 3.1 tests whether providing similar examples from other codes' variants improves both correctness and consistency through RAG (Retrieval-Augmented Generation).

How It Works
  1. Take a description variant from Chapter 3 (e.g., "Patient has profuse watery diarrhea")
  2. Find similar descriptions from our variant database (e.g., other diarrhea-related variants)
  3. Show these as examples: "Similar cases: A00 → 'severe diarrhea', A00.0 → 'watery stool'"
  4. Ask model to predict the code with this context
  5. Compare with Chapter 3's prediction (without examples)

Three Corpus Modes Tested

We test three different retrieval strategies:

  • Chapter 3.1.1 (Real Only): Search only the 46k real medical descriptions from the original dataset
  • Chapter 3.1.2 (Synthetic Only): Search only the AI-generated description variants we created
  • Chapter 3.1.3 (Both): Search both real descriptions and synthetic variants together

Two Key Performance Indicators (KPIs)

We measure:

  • Correctness (Accuracy): Does the prediction match the actual ground truth code?
  • Consistency (Stability): Do all 11 variants of the same code produce the same prediction?

Chapter 3.2: RAG-Enhanced Prediction (Billable Codes Only)

Building on Chapter 3.1: Filtering Non-Billable Codes

Chapter 3.1 demonstrated RAG-enhanced prediction across all ICD-10 codes. However, ICD-10 has a critical distinction:

  • 3-character codes (e.g., A00): Category headers - NOT billable
  • 4+ character codes (e.g., A00.0): Specific diagnoses - BILLABLE

The Key Discovery: Filtering for Billable Codes

Chapter 3.2 re-runs Chapter 3.1's exact same experiments but filters to only billable codes (LENGTH(code) > 3) using immutable SQL VIEWs. This shows whether removing non-billable category codes improves performance.

How It Works
  1. Take a description variant from Chapter 3 (e.g., "Patient has profuse watery diarrhea")
  2. Find similar descriptions from our variant database (e.g., other diarrhea-related variants)
  3. Show these as examples: "Similar cases: A00 → 'severe diarrhea', A00.0 → 'watery stool'"
  4. Ask model to predict the code with this context
  5. Compare with Chapter 3's prediction (without examples)

Three Corpus Modes Tested

We test three different retrieval strategies:

  • Chapter 3.1.1 (Real Only): Search only the 46k real medical descriptions from the original dataset
  • Chapter 3.1.2 (Synthetic Only): Search only the AI-generated description variants we created
  • Chapter 3.1.3 (Both): Search both real descriptions and synthetic variants together

Two Key Performance Indicators (KPIs)

We measure:

  • Correctness (Accuracy): Does the prediction match the actual ground truth code?
  • Consistency (Stability): Do all 11 variants of the same code produce the same prediction?

Chapter 3.3: Dense RAG Experiment (10×10 Variant Matrix)

The Density Hypothesis: More Variants = Better Retrieval

Chapter 3.1 used 11 variants per code (detail levels 0-10, one variant each). What if we dramatically increase variant density?

The 10×10 Variant Matrix Per Code

For each ICD-10 code, we generate a 10×10 matrix of variants:

  • 10 Detail Levels (rows): From ultra-concise (3-5 words) to maximum detail (40-50 words)
  • 10 Similar Variants (columns): Different phrasings at each detail level
  • Total: 100 variants per code (vs. 11 in Chapter 3.1)

With N codes, this creates a 10×10×N tensor of synthetic medical descriptions, dramatically expanding the RAG retrieval corpus.

Chapter 3.4: Dense RAG with Negative Examples

The Breakthrough Insight: Teaching What Codes Are NOT

Previous RAG experiments (Chapters 3.1-3.2) showed positive examples: "This description matches this code." But medical codes often describe similar conditions that are easily confused.

The Negative Example Hypothesis

Core Idea: Teaching the model what a code is NOT is as important as teaching what it IS.

For example, when predicting hypertension codes:

  • Positive Examples: "Essential hypertension" → I10 ✓
  • Negative Examples: "Renovascular hypertension" → I15.0 (NOT I10) ✗

By showing both, the model learns decision boundaries between easily confused codes.

Experimental Design

How It Works
  1. Take a dense variant description from Chapter 3.3
  2. Retrieve 3 positive examples: other variants from the SAME code
  3. Retrieve 2 negative examples: variants from DIFFERENT codes
  4. Present both to the model with clear labels (✓ vs ✗)
  5. Ask the model to predict the code
  6. Compare accuracy against baseline (Chapter 3.0)
